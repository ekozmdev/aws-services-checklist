# Repository Guidelines

## Project Structure & Module Organization
- 主要な成果物は `README.md` のチェックリスト表のみです。
- 余計なファイルやディレクトリは追加しません。

## Checklist Workflow
- ユーザーからコメントが来たら `README.md` の表のみを更新します。
- 表の列順は次の順番を厳守: サービス名, 知っている, 触ったことがある, メモ, 更新日。
- 「知っている」「触ったことがある」は `はい` / `いいえ` で統一します（個人利用が難しいサービスは `-` にします）。
- メモは 1〜2 文で短く、学びや感想に絞ります。
- 参考文献に紐づく場合、メモ内に参照番号（例: `[1]`）を付けます。
- サービスはカテゴリごとにまとめ、可能ならカテゴリ内でサービス名のアルファベット順に整理します。
- ユーザーが「ヒアリング」と言った場合は、対象を「そのセッションで出てきたサービス」または「知っている/触ったことがあるが `はい` なのにメモが空のサービス」に限定し、サービスを 1 つずつ挙げてメモ内容を確認するフローで進めます。
- 更新日は `yyyy/MM/dd` 形式で記入し、ステータスやメモを変更したら必ず更新します。
- 個人利用が難しいサービスは、メモに理由（料金・設備・契約など）を短く書きます。
- 表にないサービスが挙がった場合は、Web検索で存在を確認します。存在すれば該当セクションと追加位置を提案して確認を取り、存在しない場合やタイポが疑われる場合はユーザーに確認を求めます。
- ユーザーが入力したメモの内容が正しいかWeb検索を行い、ファクトチェックすること。
  - 間違っている場合は、引用元を明記して間違っている箇所を指摘すること。

## Commit & Pull Request Guidelines
- コミットメッセージは Conventional Commits を使います（例: `docs: add checklist row for EC2`）。

## Agent-Specific Instructions
- 更新対象は `README.md` の表のみです。
- 新規ファイルは作成しません。
- 更新後は行の整合性と表記ゆれを確認してから完了します。

## Web検索について

あなたが`Codex Web`上で実行されている場合、`./scripts/tavily-in-codex-web.md`を読み、その指示に従うこと。
